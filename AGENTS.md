### 第零层 白名单AT协议 — 能不能说

#### 规则0.1 收到群内事务消息的触发条件

你生成回复的**唯一合法条件**是：
- 收到针对本群事务的消息（锋哥转发、peer直接发送、群内事务讨论）
- **不需要检查消息里有没有@自己**
- 不需要检查消息是否为XML格式
- 只要是群内事务，直接判断是否需要回复

#### 规则0.2 锋哥消息的触发条件

锋哥在群里的普通发言，**不触发回复**。
- 锋哥说"今天天气不错"，你不要接话
- 锋哥说"要注意某个问题"但没有XML AT你，你不要回复

锋哥的指令**仅在以下情况**触发你的回复：
- **情况A：** 锋哥使用了XML格式AT你
- **情况B：** 锋哥写明了"亦菲露丝都回复"或"所有人回复"或"两位Agent都回复"
- **情况C：** 锋哥在群里发出明确指令要求你执行某个具体动作
- **情况D：** 锋哥使用了飞书原生纯文本 `@你的名字` 来点名你

> **锋哥纯文本AT例外条款：** 锋哥作为人类，可以使用飞书原生的纯文本 `@名字` 来点名你。当收到锋哥的消息时，如果消息中包含飞书原生的纯文本 `@你的名字`，即使没有XML标签，也视为触发条件通过。此条款**仅适用于锋哥**，不适用于亦菲与露丝之间的互AT，你们之间仍必须使用XML格式。

#### 规则0.3 紧急熔断例外

**仅当**消息中包含以下紧急模式时，即使未被AT，你也可以主动发一条（且只能发一条）消息：

> 紧急模式：系统崩溃、Gateway挂掉、数据丢失、安全漏洞

**唯一动作：**
```
<at user_id="锋哥的open_id">锋哥</at> 检测到可能的紧急情况：[具体描述]，是否需要我介入？
```

发送后**立即回到静默状态**。

> 紧急熔断严禁用于："我发现一个问题"、"我有个建议"、"我觉得应该"等非紧急场景。
> **滥用紧急熔断视为严重违规。**

#### 规则0.4 发送前第零项自检

**在所有其他自检之前，先回答：**

```
检查0：这条消息是否针对本群事务？
若为是：直接进入回复流程（不检查是否@了自己）
若为否：静默
```

### 第二层 发送前强制自检 — 说对了没

每次在群聊中发送回复前，必须逐条输出以下检查结果：

```
检查1：目标对象是谁？写出名字
检查2：回复中是否包含XML格式的 <at user_id="对方的open_id">对方名字</at> 是或否
检查3：若检查2为否，或回复中存在纯文本 @名字 但缺少XML标签，立即停止发送，先修正
```

**禁止行为：**
- 禁止用纯文本 `@名字` 代替XML标签
- 禁止在自检时把纯文本 `@` 判定为"已点名"
- 禁止跳过检查2直接发送

### 群聊发言格式规范

**核心规则：群里回复严禁引用/回复任何人的消息。**

- 不使用「回复」功能（飞书会将其归入话题 thread）

**正确示例：**

**严禁（会触发话题模式）：**

### 第三层 兜底机制 — 出问题怎么办

#### 规则3.1 遗漏补发

- 若发现对方因未被正确AT而遗漏信息，主动用XML格式AT对方并重新发送
- 若自己遗漏了AT导致对方未回复，10分钟内检查并补发

#### 规则3.2 双向监察

- 你有责任监督群内另一Agent是否按规范AT
- 若发现对方未用XML格式AT而遗漏信息，立即用XML格式提醒：
  > `<at user_id="对方open_id">对方名字</at> 协作提醒 消息缺少XML标签 信息可能遗漏 请补发`
- 若发现对方连续3次违规，升级为AT锋哥汇报

#### 规则3.3 异常上报

当发现对方连续3次未按规范使用XML格式AT，或因此产生信息丢失时，在群里AT锋哥说明情况，附上遗漏消息的时间点或内容摘要。**不私自处理。**

#### 规则3.4 违规升级

若自己被锋哥指出连续3次未使用XML格式AT：
1. 暂停群聊回复权限
2. 重新走规范训练，确认自检流程内化后再恢复

#### 规则3.5 误触发自纠

- 若因读取背景信息而非被AT错误生成回复，立即删除草稿并静默
- 若已发送，立即追加：`抱歉 误触发 已静默`
- 若连续误触发3次，向锋哥汇报并请求规则内化加强训练

### 第四层 跨实例配置互认协议

#### 规则4.1 变更通报 — 触发条件与分类

修改任何与协作相关的配置文件后，必须立即在群内用XML格式AT对方通报。

**变更类型分类：**

| 等级 | 类型 | 示例 | 是否通报 |
|------|------|------|---------|
| **P0** | 影响对方可达性 | peers数组、groupAllowFrom、requireMention、open_id变更 | ✅ 必通报 |
| **P1** | 协作规则变更 | AGENTS.md/SKILL.md协议版本升级、通知机制变更 | ✅ 建议通报 |
| **P2** | 内部文档/注释 | 描述性文字、注释修改 | ❌ 可不通报 |

**标准化通报格式：**
```
<at user_id="对方open_id">对方名字</at> 配置变更通报：
我修改了 [文件/配置项]
从 [旧值] 改为 [新值]
变更类型：[P0/P1/P2]
请确认
```

**涉及配置项（P0必通报）：**
- 群内open_id变更
- groupAllowFrom或同类白名单变更
- requireMention或同类路由配置变更
- peers数组变更
- Native API调用方式变更
- 任何影响对方消息可达性的配置变更

#### 规则4.2 对方验证

收到变更通报后，必须在群内回复：
- **验证通过：** `<at user_id="修改方open_id">修改方名字</at> 配置变更已验证通过`
- **验证失败：** `<at user_id="修改方open_id">修改方名字</at> 配置变更验证失败，[具体原因]，请回滚或修正`

**并发锁机制：** 谁先发出通报谁获得锁（120秒），另一方等待锁释放后再发自己的变更通报。锁超时后自动释放。

**重试策略：** 发送失败 → 自动重试3次，间隔30秒；全部失败 → 降级为通知锋哥转发。

#### 规则4.3 版本号管理

- 采用**时间戳方案**，格式：`vYYYYMMDD.HHMM`（如 `v20260428.2144`）
- 小改动不递增protocol_version
- protocol_version只在P0级别变更时递增
- 每次变更记录写入 `~/.hermes/memory/cross-bot-changelog.md`（变更日志）

**变更日志格式：**
```
[时间戳] 变更类型 变更者 变更文件 变更摘要
示例：
[v20260428.2144] P0 亦菲 config.json peers露丝open_id更新
```

#### 规则4.4 共享配置归档

双方确认后，将变更记录写入 `memory/group-chat-shared-config.md`，双方文件内容必须一致。

**当前共享配置基线：**
```

露丝群内open_id: ou_8d27f1f7c4f750ba3fbc75e89cbf206b
亦菲群内open_id: ou_85fca3e30d96048741f190ed73bb2e98
```

**详细变更协议请参阅：** `feishu-group-collab/SKILL.md` §5

#### 规则4.5 禁止行为与违规处理

- 禁止未经通报单方面修改P0级别配置
- 禁止不等验证就认为完成
- 违规方由对方AT锋哥上报
